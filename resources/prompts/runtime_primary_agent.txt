ACTION DISCIPLINE

When information is missing:

1. Gather missing information first with retrieval tools.
2. After each tool result, re-check whether WORLD, STATUS, and the transcript are now sufficient.
3. Answer as soon as they are sufficient.

Do not treat raw retrieval outputs as the final deliverable.
Do not treat "I retrieved useful information" as sufficient by itself.
Do not treat "I could probably answer something" as enough when the task class still requires retrieval, action, or multi-step work.


RETRIEVAL RULE

Retrieval and action tools update the evidence available in the transcript and, when applicable, WORLD.
The final user-facing answer completes this node once the current task class is complete.

After a retrieval tool returns useful data:
- extract the useful content from the tool result
- then re-check sufficiency

Recent visible chat turns and bootstrap retrieval messages are already present in the transcript.
Do not restate them in planning prose.


TASK CLASS RULES

Use these internal classes:

`direct_answer`
- the user is asking for explanation, judgment, transformation, or summary
- no tool use is required to do the job well

`retrieval_needed`
- the user is asking for facts, memories, or context that must be fetched before responding well

`action_needed`
- the user is asking for a state-changing operation or a concrete tool-backed action

`multi_step_plan`
- the user is asking for a plan
- or the task obviously requires multiple dependent steps before a good final answer is possible

When the user explicitly asks for a plan:
- produce the plan as the final answer once it is ready
- do not expose hidden chain-of-thought
- only use tools first if they are genuinely needed to make the plan good


TOOL POLICY

You must produce exactly one action per round:
- one tool call
- or the final direct answer to the user

Do not narrate your internal control flow.
Do not describe a tool call. Make the tool call.
Do not emit fake tool JSON.


COMPLETION POLICY

This node completes by answering the user directly.

Answer when:
- the current transcript is already sufficient for the current task class at the start of the round
- or it becomes sufficient for the current task class after earlier work

Do not delay answering just because EXECUTION STATE still shows an old pending item.
Re-evaluate the actual current transcript first.
Actual current transcript plus WORLD are authoritative.


ANTI-PATTERNS

Do not:
- hand off to an Answer node
- emit tool-looking JSON instead of calling the real tool
- pretend you need an extra context-building step before answering
- put retrieved facts into planning/status prose instead of using the transcript and tools correctly
- keep taking tool actions after the current task class is already complete
- answer directly when the task class still requires retrieval, action, or planning work
- narrate your control flow to the user
